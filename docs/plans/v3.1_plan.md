# V3.1 Implementation Plan — Tyre Performance Curves, Pit Lane Flow & Degradation Cliff

## Context

V3 shipped tyre compounds, pit stops, fuel, and stint tracking. But three areas need correction:

1. **Tyre pace differences are too coarse** — Current `peak_multiplier` values (soft=1.05, medium=1.0, hard=0.95) create a ~2.4s/lap gap between soft and hard. The user wants precise deltas: Medium = +0.3s/lap vs soft, Hard = +0.6s/lap vs soft.

2. **Pit stop is a "freeze + teleport"** — The car's `distance_along_track` freezes at `pit_entry_distance` during the entire pit stop, then snaps to `pit_exit_distance` on exit. The car should visibly travel at reduced speed through the pit lane during entry and exit phases.

3. **Degradation is purely linear** — Current model: `multiplier = peak - (laps * rate)` with a hard floor. The user wants a realistic tyre life curve with three phases: peak window, gradual falloff, and a dangerous cliff below 30%.

## Change 1: Tyre Compound Pace Tuning

### Problem

The current compound config uses `peak_multiplier` to differentiate pace:
- Soft: `peak_multiplier = 1.05`
- Medium: `peak_multiplier = 1.0`
- Hard: `peak_multiplier = 0.95`

On Monza (~2025.5 units, v_ref 83 u/s), one lap takes ~24.4s. A 5% multiplier gap between soft and medium = ~1.22s/lap. The user wants:
- **Medium = +0.3s/lap compared to soft**
- **Hard = +0.6s/lap compared to soft**

### Solution: Config-Only Change (No Code Modifications)

The `peak_multiplier` mechanism already works correctly — it directly scales speed. We just need the right values.

**Math**: Lap time = track_length / (v_ref × peak_multiplier × other_multipliers). At peak (no degradation penalty, fresh tyres), the "other_multipliers" average ~1.0 across a full lap (varies by corner). So:

```
lap_time ≈ track_length / (v_ref × peak_multiplier)

For soft (baseline):  T_soft = 2025.5 / (83.0 × P_soft)
For medium: T_medium = T_soft + 0.3
For hard:   T_hard = T_soft + 0.6
```

If we set soft `peak_multiplier = 1.0` (baseline, no boost):
- T_soft ≈ 24.40s
- T_medium = 24.70s → P_medium = 2025.5 / (83.0 × 24.70) ≈ 0.9878... → **0.988**
- T_hard = 25.00s → P_hard = 2025.5 / (83.0 × 25.00) ≈ 0.9761... → **0.976**

## Change 2: Pit Lane Distance-Based Movement

### Problem

Currently in [pit_stop_manager.gd](game/sim/src/pit_stop_manager.gd):
- **ENTRY phase**: `car.distance_along_track` freezes at `pit_entry_distance` (1950). A countdown timer runs for `pit_entry_duration` (8s). The car has `effective_speed = pit_lane_speed` (20 u/s) set on it, but this speed is never applied to actual movement — Phase 3 of the step loop skips pit cars.
- **STOPPED phase**: Same frozen position. Timer runs.
- **EXIT phase**: Same frozen position. Timer runs. On completion, position snaps to `pit_exit_distance` (80).

The car effectively "freezes and teleports". The user wants: car enters at reduced speed, travels to pit box, stops, then exits at reduced speed to the rejoin point.

### Solution: Compute Durations From Distance, Interpolate Position

Replace the config-driven `pit_entry_duration` and `pit_exit_duration` with durations computed from the actual pit lane distance and speed. During entry/exit, interpolate `distance_along_track` so the car visibly moves.

**Pit lane geometry on Monza**:

```
Track: ---[pit_entry=1950]===S/F LINE(~2025.5)===0===[pit_exit=80]---
                \                                      /
                 \--------- PIT LANE ----------------/
```

The pit lane runs from entry (1950) forward past the S/F line to exit (80). The "pit lane distance" wrapping around the track boundary is:

```
pit_lane_distance = (track_length - pit_entry_distance) + pit_exit_distance
                  = (2025.5 - 1950) + 80 = 155.5 units
```

We define a **pit box position** at the midpoint of this pit lane distance. The car travels from entry to pit box (entry phase), stops, then travels from pit box to exit (exit phase).

```
entry_distance = pit_lane_distance / 2 = 77.75 units
exit_distance  = pit_lane_distance / 2 = 77.75 units

entry_duration = entry_distance / pit_lane_speed = 77.75 / 20 = 3.89 seconds
exit_duration  = exit_distance / pit_lane_speed = 77.75 / 20 = 3.89 seconds
```

During these phases, `distance_along_track` is interpolated (wrapping around the S/F line).

### New Config Field: `pit_box_distance`

Add an optional `pit_box_distance` field to `PitConfig`. This is the distance along the main track where the pit box is located. If not provided, it defaults to the midpoint between entry and exit (wrapping forward).

For Monza: midpoint of pit lane = entry + half pit lane distance = 1950 + 77.75 = 2027.75 → wraps to 2027.75 - 2025.5 ≈ 2.25. So pit box is at ~2.25 units (just past S/F line).

But for configurability (different tracks may have the pit box closer to entry or exit), make it explicit:

```json
"pit": {
  "pit_box_distance": 2.0
}
```

Add to `clone()`.

```gdscript
func process_pit_phase(car: RaceTypes.CarState, chunk_dt: float) -> Dictionary:
    # ... existing guards ...

    # Interpolate position during ENTRY and EXIT phases BEFORE decrementing time
    if car.pit_phase == RaceTypes.PitPhase.ENTRY:
        var entry_duration: float = _entry_distance / maxf(_config.pit_lane_speed_limit, 0.001)
        var progress: float = 1.0 - (car.pit_time_remaining / maxf(entry_duration, 0.001))
        progress = clampf(progress, 0.0, 1.0)
        car.distance_along_track = _lerp_track_distance(
            _config.pit_entry_distance, _pit_box_distance, progress, _track_length)

    elif car.pit_phase == RaceTypes.PitPhase.EXIT:
        var exit_duration: float = _exit_distance / maxf(_config.pit_lane_speed_limit, 0.001)
        var progress: float = 1.0 - (car.pit_time_remaining / maxf(exit_duration, 0.001))
        progress = clampf(progress, 0.0, 1.0)
        car.distance_along_track = _lerp_track_distance(
            _pit_box_distance, _config.pit_exit_distance, progress, _track_length)

    car.pit_time_remaining = maxf(car.pit_time_remaining - chunk_dt, 0.0)
    if car.pit_time_remaining > 0.0:
        return default_result

    # Phase transitions:
    if car.pit_phase == RaceTypes.PitPhase.ENTRY:
        car.pit_phase = RaceTypes.PitPhase.STOPPED
        car.pit_time_remaining = compute_stop_duration(car)
        car.distance_along_track = _pit_box_distance  # At the box
        return default_result

    if car.pit_phase == RaceTypes.PitPhase.STOPPED:
        # ... tyre change + refuel (existing logic) ...
        car.pit_phase = RaceTypes.PitPhase.EXIT
        var exit_duration: float = _exit_distance / maxf(_config.pit_lane_speed_limit, 0.001)
        car.pit_time_remaining = exit_duration
        car.distance_along_track = _pit_box_distance  # Start exit from box
        return { "completed": false, "new_compound": car.pit_target_compound, "refueled_to_kg": fuel_after_stop }

    if car.pit_phase == RaceTypes.PitPhase.EXIT:
        # ... existing rejoin logic ...
        car.distance_along_track = _config.pit_exit_distance
        return { "completed": true, ... }
```

**New helper**: `_lerp_track_distance()` — interpolate along the track with wrapping:

```gdscript
func _lerp_track_distance(from_d: float, to_d: float, t: float, track_length: float) -> float:
    var forward_dist: float = fposmod(to_d - from_d, track_length)
    return fposmod(from_d + forward_dist * clampf(t, 0.0, 1.0), track_length)
```

## Change 3: Non-Linear Degradation Curve with Cliff

### Problem

Current `DegradationModel.compute_multiplier()` in [degradation_model.gd](game/sim/src/degradation_model.gd) is purely linear:

```
multiplier = peak_multiplier - (laps_after_warmup × degradation_rate)
```

Clamped to `[min_multiplier, peak_multiplier]`.

The user wants a three-phase tyre life curve:

| Tyre Life % | Phase | Behavior |
|---|---|---|
| 100% → 75% | Peak window | Optimal performance, no degradation |
| 75% → 30% | Gradual falloff | Very small reduction, slow decline |
| < 30% | Danger zone | Rapid degradation, risk of puncture |

### Defining "Tyre Life Percentage"

Tyre life is how much performance remains before hitting the floor:

```
tyre_life = (current_multiplier - min_multiplier) / (peak_multiplier - min_multiplier)
```

At peak: `tyre_life = 1.0` (100%)
At floor: `tyre_life = 0.0` (0%)

### New Degradation Curve Model

Replace the linear degradation with a **piecewise curve** that has three phases. The curve is defined by two thresholds (as tyre life percentages) and a cliff acceleration factor:

```
optimal_threshold: float = 0.75    # above this = peak performance (no degradation)
cliff_threshold: float = 0.30      # below this = danger zone (rapid degradation)
cliff_multiplier: float = 3.0      # how much faster degradation is in the cliff zone
```

**The curve works as follows:**

Given a car's stint progress (stint_laps after warmup), compute effective degradation:

1. **Phase 1 (Optimal)**: While `tyre_life >= optimal_threshold` (75%):
   - `multiplier = peak_multiplier` (no loss — tyres are in their optimal window)

2. **Phase 2 (Gradual)**: While `optimal_threshold > tyre_life >= cliff_threshold` (75% → 30%):
   - Linear degradation at the configured `degradation_rate`
   - Slow, predictable decline

3. **Phase 3 (Cliff)**: While `tyre_life < cliff_threshold` (< 30%):
   - Degradation rate accelerates by `cliff_multiplier` (e.g., 3× faster)
   - `effective_rate = degradation_rate × cliff_multiplier`
   - Multiplier drops rapidly toward `min_multiplier`

### Implementation in [degradation_model.gd](game/sim/src/degradation_model.gd)

**New config fields** on `DegradationConfig` in [race_types.gd](game/sim/src/race_types.gd):

```gdscript
var optimal_threshold: float = 1.0     # tyre life % above which no degradation occurs
var cliff_threshold: float = 0.0       # tyre life % below which degradation accelerates
var cliff_multiplier: float = 1.0      # degradation rate multiplier in cliff zone
```

**IMPORTANT**: Defaults are `1.0, 0.0, 1.0` which produce the exact same linear behavior as before.
When `optimal_threshold = 1.0`, Phase 1 has zero width → degradation starts immediately.
When `cliff_multiplier = 1.0`, Phase 3 has the same rate as Phase 2 → no cliff.

**Modified `compute_multiplier()`:**

```gdscript
static func compute_multiplier(
    lap_count: int,
    fractional_lap: float,
    config: DegradationConfig
) -> float:
    if config == null:
        return 1.0

    var clamped_fractional_lap: float = clampf(fractional_lap, 0.0, 0.999999)
    var race_progress: float = maxf(float(lap_count), 0.0) + clamped_fractional_lap
    var warmup_laps: float = maxf(config.warmup_laps, 0.0)
    var peak: float = config.peak_multiplier
    var floor_mult: float = minf(config.min_multiplier, peak)
    var rate: float = maxf(config.degradation_rate, 0.0)

    # Warmup phase (UNCHANGED from V3)
    var warmup_start: float = maxf(peak - warmup_laps * rate, floor_mult)
    if warmup_laps > 0.0 and race_progress < warmup_laps:
        var t: float = race_progress / warmup_laps
        return clampf(lerpf(warmup_start, peak, t), floor_mult, peak)
    if race_progress <= warmup_laps:
        return clampf(peak, floor_mult, peak)

    # Post-warmup: three-phase degradation
    var laps_after_warmup: float = race_progress - warmup_laps
    var total_range: float = peak - floor_mult
    if total_range <= 0.0 or rate <= 0.0:
        return clampf(peak, floor_mult, peak)

    var optimal_thresh: float = clampf(config.optimal_threshold, 0.0, 1.0)
    var cliff_thresh: float = clampf(config.cliff_threshold, 0.0, optimal_thresh)
    var cliff_mult: float = maxf(config.cliff_multiplier, 1.0)

    # Convert thresholds to multiplier values
    var optimal_mult: float = floor_mult + total_range * optimal_thresh
    var cliff_mult_value: float = floor_mult + total_range * cliff_thresh

    # Phase 1: Optimal — no degradation until multiplier would drop below optimal_mult
    var optimal_laps: float = (peak - optimal_mult) / rate if rate > 0.0 else INF
    if laps_after_warmup <= optimal_laps:
        return clampf(peak, floor_mult, peak)

    # Phase 2: Gradual — normal rate from optimal_mult down to cliff_mult_value
    var gradual_laps: float = (optimal_mult - cliff_mult_value) / rate if rate > 0.0 else INF
    var laps_into_gradual: float = laps_after_warmup - optimal_laps
    if laps_into_gradual <= gradual_laps:
        var degraded: float = optimal_mult - laps_into_gradual * rate
        return clampf(degraded, floor_mult, peak)

    # Phase 3: Cliff — accelerated rate from cliff_mult_value down to floor
    var laps_into_cliff: float = laps_into_gradual - gradual_laps
    var cliff_rate: float = rate * cliff_mult
    var degraded: float = cliff_mult_value - laps_into_cliff * cliff_rate
    return clampf(degraded, floor_mult, peak)
```

### Example: Soft Compound Trajectory

Config: peak=1.0, min=0.70, rate=0.04, optimal=0.75, cliff=0.30, cliff_mult=3.0

```
total_range = 1.0 - 0.7 = 0.3
optimal_mult = 0.7 + 0.3 × 0.75 = 0.925
cliff_mult_value = 0.7 + 0.3 × 0.30 = 0.79

Phase 1 (Optimal): peak(1.0) to 0.925 = 0.075 range / 0.04 rate = 1.875 laps
  Laps 0-1.875: multiplier = 1.0 (peak, no loss)

Phase 2 (Gradual): 0.925 to 0.79 = 0.135 range / 0.04 rate = 3.375 laps
  Lap 1.875: 0.925
  Lap 3.25:  0.870
  Lap 5.25:  0.790

Phase 3 (Cliff): 0.79 to 0.70 = 0.09 range / (0.04×3=0.12) = 0.75 laps
  Lap 5.25:  0.790
  Lap 5.625: 0.745
  Lap 6.0:   0.700 (floor)

Total tyre life: 1.875 + 3.375 + 0.75 = 6.0 laps before floor
```

Compare to old linear model: `(1.0 - 0.7) / 0.04 = 7.5 laps` — but it was losing performance from lap 0. The new model gives 1.875 laps of FULL performance, then gradual loss, then a cliff.

### Example: Hard Compound Trajectory

Config: peak=0.976, min=0.80, rate=0.01, optimal=0.75, cliff=0.30, cliff_mult=3.0

```
total_range = 0.976 - 0.80 = 0.176
optimal_mult = 0.80 + 0.176 × 0.75 = 0.932
cliff_mult_value = 0.80 + 0.176 × 0.30 = 0.8528

Phase 1 (Optimal): 0.976 to 0.932 = 0.044 / 0.01 = 4.4 laps of peak
Phase 2 (Gradual): 0.932 to 0.8528 = 0.0792 / 0.01 = 7.92 laps
Phase 3 (Cliff): 0.8528 to 0.80 = 0.0528 / 0.03 = 1.76 laps

Total: ~14 laps — hard tyres last much longer and have 4.4 laps of full peak!
```

### Puncture Risk — Deferred

The user mentioned puncture risk at < 30%. For V3.1, we implement the cliff acceleration (rapid degradation below 30%). Actual puncture events (random tire failure) are deferred to a future version because:
1. Punctures require a new event system (car forced to pit immediately)
2. Randomness needs the seed-based RNG system
3. The cliff degradation already punishes staying out too long — performance craters

The cliff zone makes it strategically terrible to stay out below 30%, which achieves the user's intent without the complexity of a random puncture system.

### Backward Compatibility

Default values (`optimal_threshold=1.0, cliff_threshold=0.0, cliff_multiplier=1.0`) produce the **exact same linear behavior** as before:
- `optimal_threshold=1.0` → Phase 1 has zero width (peak to 100% of range = 0 laps) → degradation starts immediately
- `cliff_multiplier=1.0` → Phase 3 rate = normal rate → no cliff acceleration

**V1/V1.1/V2 configs**: These don't set the new fields → get defaults → identical behavior. Zero breakage.

**V3 configs**: Must explicitly add `optimal_threshold`, `cliff_threshold`, `cliff_multiplier` to each compound's degradation to activate the curve.

### Guardrails

- **DO NOT change the warmup phase.** It stays exactly the same (linear interpolation from warmup_start to peak). Only post-warmup behavior changes.
- **DO NOT break backward compatibility.** Default values MUST produce the old linear behavior. New curve only activates when fields are explicitly set in config.
- **DO NOT implement puncture events in V3.1.** The cliff degradation provides the gameplay incentive to pit before 30%. Punctures are a future feature.
- **DO NOT change how `stint_tracker.gd` feeds inputs to the model.** The `(stint_lap_count, fractional_lap, config)` interface stays the same. Only the internal curve shape changes.
- **DO NOT use the degradation thresholds for HUD display.** The HUD shows `degradation_multiplier` as a percentage. It does NOT show "tyre life %". That mapping (if desired) is a separate future HUD enhancement.
