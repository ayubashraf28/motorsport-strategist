# ADR 0003: V1.1 Physics-Derived Speed Profile

## Status

Accepted

## Context

V1 used hand-authored pace segments with per-segment multipliers to vary car speed around the
track. This works but requires manual tuning per track and per corner, does not generalise to
new tracks, and encodes no physical reasoning.

V1.1 needs:
- speed variation derived from track geometry (curvature) rather than hand-authored values
- realistic braking zones upstream of corners and acceleration out of them
- deterministic, testable behaviour
- config-driven vehicle parameters (grip, accel, braking)
- real track geometry as a reusable data asset

## Decision

### 1. Track geometry from TUMFTM racetrack-database

Use the TUMFTM/racetrack-database (LGPL-3.0) as the source of truth for Monza's centerline.
The raw CSV is processed by an offline Python script (`tools/scripts/import_track.py`) into
a committed JSON asset (`data/tracks/monza/monza_centerline.json`) containing uniformly-spaced
sample points with pre-computed signed curvature values.

The raw CSV is excluded from git. Only the derived JSON asset is committed. Attribution is
documented in `data/tracks/monza/README.md`.

### 2. Curvature computed at import time, stored in the asset

Curvature is computed in the Python import script using the three-point signed formula and
stored in the JSON asset. Godot reads the pre-computed values; it does not recompute curvature
at runtime. This keeps runtime startup deterministic and fast, and makes curvature tunable
by rerunning the import script.

### 3. Level B physics model (quasi-steady-state)

Corner speed is limited by lateral acceleration:

```
v_max_corner(s) = min(sqrt(a_lat_max / max(|κ(s)|, ε)), v_top_speed)
```

Longitudinal constraints are applied via a forward pass (acceleration) and backward pass
(braking), each iterated twice around the circular track to satisfy the loop boundary condition:

```
Forward:  v_fwd[i+1] = min(v_corner[i+1], sqrt(v_fwd[i]² + 2·a_long_accel·ds))
Backward: v_bwd[i-1] = min(v_corner[i-1], sqrt(v_bwd[i]² + 2·a_long_brake·ds))
Final:    v[i]       = min(v_corner[i], v_fwd[i], v_bwd[i])
```

All vehicle parameters (`a_lat_max`, `a_long_accel`, `a_long_brake`, `v_top_speed`) live in
`config/race_v1.1.json` under a `physics` block. No values are hardcoded.

### 4. Per-car scaling via v_ref

The physics profile gives the theoretical maximum speed at each track position. Each car scales
this profile by `v_ref / v_top_speed`, where `v_ref` is the car's reference speed (the speed it
would travel on a flat straight at peak performance). Cars with lower `v_ref` are uniformly
slower across the whole lap — they still follow the same cornering shape, just at a reduced scale.

This is equivalent to V1's `base_speed * multiplier` except the multiplier is now computed from
physics rather than hand-authored, and `v_ref` plays the role `base_speed` played in V1.

`current_multiplier` in `CarState` is retained as `effective_speed / v_top_speed` for HUD
display. No HUD changes are required.

### 5. Geometry data crosses the sim/presentation boundary via RaceRuntimeParams

`TrackGeometryData` (sample positions and curvatures) is loaded by `RaceController` (presentation
layer) via `TrackLoader` and passed to `RaceSimulator.initialize()` through `RaceRuntimeParams`.
The simulator never performs file I/O. The sim/presentation boundary is preserved.

### 6. Schema version routing in RaceConfigLoader

`race_v1.json` (schema version absent or `"1.0"`) continues to work via the existing parse path.
`race_v1.1.json` (`schema_version: "1.1"`) routes to a new parse branch that produces
`SpeedProfileConfig` instead of `PaceProfileConfig`. The simulator branches on the type of
`RaceConfig.track` at initialisation time.

V1 tests and the V1 runtime path are untouched.

## Alternatives considered

### Keep hand-authored multipliers, add more segments

Pros:
- zero new complexity

Cons:
- does not scale to new tracks
- manual tuning for every corner on every track
- no physical basis for values

### Full vehicle dynamics (rigid body, tyre contact patch)

Pros:
- highest physical fidelity

Cons:
- significant complexity, not proportionate to V1.1 goals
- determinism is harder to guarantee with contact-patch models
- outside scope — planned for a later version if needed

### Compute curvature at Godot runtime from raw polyline

Pros:
- single data source (no import step)

Cons:
- startup cost paid every session
- harder to unit-test (requires Godot runtime)
- floats may differ across machines depending on GDScript float precision

Pre-computing at import time is deterministic across all machines and keeps the Godot sim layer
free of geometry preprocessing logic.

## Consequences

- V1.1 speed profiles are physically motivated and generalise to any track importable from TUMFTM.
- Vehicle parameters are fully config-driven; tuning requires only a JSON edit.
- The sim layer remains pure GDScript logic with no file I/O or Godot Node dependencies.
- V1 remains fully functional via schema version routing.
- Adding new tracks requires running `import_track.py` and committing the derived JSON asset.
